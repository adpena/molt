# fuzz_tir_generated_opcode_shapes

## Summary

Moved `runtime/molt-backend/fuzz/fuzz_targets/fuzz_tir_passes.rs` off its
local opcode palette, operand-count match, and result-count `matches!` hand-set.
The fixed-shape fuzz palette now lives in `runtime/molt-tir/src/tir/op_kinds.toml`
as `fuzz_tir_opcode_shapes`, renders to `FUZZ_TIR_OPCODE_SHAPES`, and uses the
existing `opcode_fixed_result_count_table` for result allocation.

`Copy` was deliberately excluded because the registry marks it
`result_arity = "variable"`. The generator rejects variable/two-result opcodes
from this fixed-shape fuzz palette rather than letting the harness encode a
private result rule.

The same pass repaired the fuzz target's stale `passes::run_pipeline` call by
passing `TargetInfo::native_release_fast()`, matching the live TIR pipeline
contract. `runtime/molt-backend/fuzz/Cargo.lock` was refreshed because the fuzz
workspace lock was stale against current manifests (`molt-tir`, Cranelift
0.131, Windows deps, `sha2`, etc.); locked compile now succeeds.
