# WebGPU Browser Inference Roadmap

## 1. Current State: CPU-Only WASM

The browser inference path loads a 3.9 MB WASM module (the molt-compiled
Falcon-OCR runtime) and 129 MB of INT4-quantized weights from R2/CDN cached
in IndexedDB. All computation runs on the CPU via the WASM linear memory
interpreter. On a 2024 MacBook Pro (M3), inference takes ~4s per page. On a
mid-range Android device, it takes 15-30s.

Matmul dominates the compute profile. For a single forward pass through
Falcon-OCR's transformer layers, approximately 90% of wall-clock time is
spent in matrix multiplication (general matrix multiply, GEMM). The remaining
10% is elementwise ops (LayerNorm, GELU, softmax) which are memory-bound and
already fast enough on CPU.

## 2. WebGPU Dispatch Architecture

The goal: keep the WASM module as the control plane (token management, KV
cache bookkeeping, output decoding) but offload matmul and other
compute-heavy operations to the GPU via WebGPU.

### 2.1 JS Import Bridge

WASM cannot call WebGPU APIs directly. The bridge is a set of JS functions
imported into the WASM module at instantiation:

```
WASM module
  │
  ├─ import "env"."gpu_buffer_create"(size: u32) -> handle: u32
  ├─ import "env"."gpu_buffer_write"(handle: u32, ptr: u32, len: u32)
  ├─ import "env"."gpu_buffer_read"(handle: u32, ptr: u32, len: u32)
  ├─ import "env"."gpu_matmul"(a: u32, b: u32, out: u32, m: u32, k: u32, n: u32)
  ├─ import "env"."gpu_elementwise"(shader_id: u32, bufs: u32, n_bufs: u32, n: u32)
  └─ import "env"."gpu_sync"()
```

The JS glue layer implements these imports by:
1. Maintaining a handle table mapping u32 handles to GPUBuffer objects.
2. Using `device.queue.writeBuffer()` to copy data from WASM linear memory
   to GPU buffers.
3. Dispatching pre-compiled compute pipelines for matmul and elementwise ops.
4. Using `mapAsync()` + `getMappedRange()` to copy results back to WASM
   linear memory when `gpu_sync()` or `gpu_buffer_read()` is called.

### 2.2 Pipeline Pre-Compilation

On initialization (after `navigator.gpu.requestAdapter()` succeeds), the JS
layer pre-compiles a fixed set of WGSL compute shaders:

- **Matmul**: Tiled matmul with configurable tile size (16x16 default).
  Generated by the molt-gpu `WgslRenderer` from a fused MUL+REDUCE_SUM
  kernel.
- **LayerNorm**: Fused mean + variance + normalize + scale + bias.
- **GELU**: Elementwise approximation `0.5 * x * (1 + tanh(sqrt(2/pi) * (x + 0.044715 * x^3)))`.
- **Softmax**: Fused max-reduce + exp + sum-reduce + div.

Pre-compilation takes ~50ms on Chrome (SPIR-V path) and ~30ms on Safari
(Metal path). Pipelines are cached for the session lifetime.

### 2.3 Data Flow

```
CPU (WASM)                          GPU (WebGPU)
─────────                           ────────────
Token embedding lookup
  │
  ▼
Write Q, K, V to GPU buffers ──►  gpu_buffer_write()
                                       │
                                       ▼
                                  matmul(Q, K^T) ──► attention scores
                                  softmax(scores)
                                  matmul(scores, V) ──► attention output
                                  matmul(attn, W_o) ──► projected output
                                       │
                                       ▼
Read output back to WASM ◄──────  gpu_buffer_read()
  │
  ▼
LayerNorm (CPU or GPU)
  │
  ▼
FFN matmul ──► gpu_matmul()
  │
  ▼
Output logits, argmax (CPU)
```

For maximum throughput, the GPU operations within a transformer layer are
chained without CPU readback between them. Only the final layer output is
read back. This minimizes the GPU-CPU synchronization barrier, which is the
primary bottleneck in WebGPU inference (see section 4).

## 3. Operations That Benefit From GPU

| Operation | % of compute | GPU speedup | Priority |
|-----------|-------------|-------------|----------|
| Matmul (GEMM) | ~90% | 10-100x | P0 |
| Softmax | ~3% | 2-5x | P1 |
| LayerNorm | ~2% | 2-3x | P2 |
| GELU/activation | ~2% | 2-3x | P2 |
| Embedding lookup | ~1% | 1x (memory-bound) | Skip |
| Argmax/sampling | <1% | 1x (trivial) | Skip |

Matmul is the only operation where GPU dispatch is unambiguously worth the
overhead of buffer transfer. For small elementwise ops, the GPU dispatch
latency (24-71 us per Maczan 2026) can exceed the CPU compute time.

The crossover point: operations on tensors smaller than ~4096 elements are
faster on CPU. Operations on tensors larger than ~16384 elements are faster
on GPU. The 4K-16K range depends on the specific backend (Vulkan is faster
to dispatch than Metal).

## 4. Expected Speedup

### Matmul-Only GPU Dispatch (P0)

Replacing only the matmul operations with WebGPU compute:

| Device | CPU-only | Matmul on GPU | Speedup |
|--------|----------|---------------|---------|
| M3 MacBook Pro | 4.0s | 0.4-0.8s | 5-10x |
| Snapdragon 8 Gen 3 | 18s | 2-4s | 4.5-9x |
| Intel i7-1365U (iGPU) | 12s | 1.5-3s | 4-8x |
| Apple A17 Pro (iPhone) | 8s | 0.8-1.5s | 5-10x |

### Full GPU Dispatch (P0 + P1 + P2)

Additionally moving softmax, LayerNorm, and GELU to GPU:

| Device | Matmul-only GPU | Full GPU | Additional speedup |
|--------|----------------|----------|-------------------|
| M3 MacBook Pro | 0.6s | 0.3-0.5s | 1.2-2x |
| Snapdragon 8 Gen 3 | 3s | 1.5-2.5s | 1.2-2x |

The diminishing returns beyond matmul are because:
1. Elementwise ops are memory-bound, not compute-bound.
2. Each additional GPU dispatch adds 24-71 us of overhead.
3. Small tensors may be faster on CPU than GPU.

### Bottleneck: GPU-CPU Synchronization

The dominant cost after GPU matmul is the synchronization barrier when
reading results back from GPU to CPU. `mapAsync()` + `getMappedRange()` adds
0.1-1ms per readback depending on buffer size and browser. Minimizing the
number of GPU-CPU round trips per forward pass is critical.

Target: 1 GPU-CPU sync per transformer layer (read back only the layer
output), not 1 per operation.

## 5. Browser Compatibility

| Browser | WebGPU | Compute Shaders | Subgroups | Status |
|---------|--------|-----------------|-----------|--------|
| Chrome 113+ | Yes | Yes | Chrome 134+ | Full support |
| Edge 113+ | Yes | Yes | Edge 134+ | Full support (Chromium) |
| Safari 18+ | Yes | Yes | No | Compute works, no subgroups |
| Firefox | Behind flag | Behind flag | No | Not production-ready |
| Chrome Android 121+ | Yes | Yes | No | Works on Adreno/Mali GPUs |
| Safari iOS 18+ | Yes | Yes | No | Works on A15+ chips |

### Fallback Strategy

```javascript
async function selectBackend() {
    if (navigator.gpu) {
        const adapter = await navigator.gpu.requestAdapter();
        if (adapter) {
            const device = await adapter.requestDevice();
            return new WebGPUBackend(device);  // GPU matmul
        }
    }
    return new CPUBackend();  // WASM-only fallback
}
```

The WASM CPU path remains the universal fallback. The WebGPU path is a
progressive enhancement that activates only when the browser supports it and
a GPU adapter is available. Feature detection is synchronous
(`navigator.gpu` exists) but adapter acquisition is async and can fail on
systems with no discrete or integrated GPU.

## 6. Implementation Phases

### Phase 1: WebGPU Matmul PoC (This PR)

- `deploy/browser/webgpu-matmul.js`: Standalone matmul using WGSL from
  WgslRenderer.
- Validates correctness against CPU reference implementation.
- Benchmarks GPU vs. CPU matmul for representative sizes (512x512, 1024x1024,
  2048x2048).

### Phase 2: JS Import Bridge

- Define the `env.gpu_*` import table in the WASM instantiation glue.
- Implement handle-table buffer management.
- Wire matmul dispatch through the bridge.
- Integration test: single transformer layer with GPU matmul.

### Phase 3: Full Inference Pipeline

- Pre-compile all shader pipelines on init.
- Chain GPU dispatches within a transformer layer (no intermediate readback).
- Progressive enhancement: detect WebGPU, fall back to CPU.
- Benchmark end-to-end inference on target devices.

### Phase 4: Optimization

- Tune tile sizes per GPU vendor (Adreno prefers 8x8, Apple GPU prefers
  16x16, desktop prefers 16x16 or 32x32).
- Enable WebGPU subgroups for reduction ops where available (Chrome 134+).
- Investigate `GPUBuffer` import from WASM shared memory to eliminate
  copies (speculative, no browser supports this yet).
- Batch command encoding per Maczan 2026 for non-autoregressive workloads.
