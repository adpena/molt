# Molt CI Plan (GitHub Actions)

Minimum required pre-merge gates are defined in
`docs/spec/areas/testing/0008_MINIMUM_MUST_PASS_MATRIX.md`.

Status: Active
Last updated: 2026-05-20

## 1. Pipeline Overview
Molt uses a tiered GitHub Actions model:

- local machine = full authoritative gatekeeper;
- push / pull_request CI = cheap required presubmit only;
- nightly CI = correctness-heavy hosted automation;
- release and perf workflows = targeted hosted validation.

## 1.1 Current Hosted Workflow Classes

| Workflow | Trigger | Purpose | Expected Cost |
| --- | --- | --- | --- |
| `ci.yml` | `push`, `pull_request` | cheap required presubmit | low |
| `nightly.yml` | `schedule`, `workflow_dispatch` | correctness-heavy automation | medium/high |
| `release.yml` | tag push, `workflow_dispatch` | hosted cross-platform release validation and publication | high |
| `perf-validation.yml` | `workflow_dispatch` | manual benchmark validation | high |

Required CI policy notes:
- Local developer verification is the authoritative full gatekeeper: run `tools/dev.py lint` and `tools/dev.py test` before landing or cutting releases.
- Use `python3 tests/harness/run_molt_conformance.py --suite smoke` for cheap local correctness checks and `--suite full` before heavier correctness work or release prep.
  The direct runner validates the current checkout by default (`sys.executable -m molt.cli`), only uses an installed binary when `MOLT_BIN` is set explicitly, and preserves an ambient `MOLT_SESSION_ID` so concurrent local runs keep isolated build state.
- GitHub Actions remains required for cheap push-time confidence plus hosted build/release validation across supported environments; do not remove release-relevant hosted lanes just because local verification exists.
- The dedicated `docs-gates` job owns documentation architecture enforcement and generated-doc freshness checks.
- `ci.yml` should stay Linux-first and should not carry full differential, full benchmark, or broad macOS release validation.
- `nightly.yml` owns the heavy correctness lanes: canonical Molt conformance and the curated differential basic/stdlib lane.
- `release.yml` owns explicit macOS and Linux release validation plus artifact publication checks.
- `perf-validation.yml` owns generic benchmark execution outside push-time CI and binds explicitly to the release-built Molt binary.
- Differential jobs should pass `--build-profile dev` explicitly; release-oriented diff or parity investigations should pass `--build-profile release` explicitly.
- Differential jobs collect RSS metrics and enforce adaptive process/tree/global memory budgets by default. Do not pin `MOLT_DIFF_RLIMIT_GB` in CI unless an investigation intentionally needs a narrower cap.
- `tools/ci_gate.py` seeds canonical repo-local artifact roots for child
  checks (`MOLT_EXT_ROOT`, `CARGO_TARGET_DIR`, `MOLT_DIFF_CARGO_TARGET_DIR`,
  `MOLT_CACHE`, `MOLT_DIFF_ROOT`, `MOLT_DIFF_TMPDIR`, `UV_CACHE_DIR`, and
  `TMPDIR`) without overriding explicit caller settings. It also defaults
  `MOLT_SESSION_ID=ci-gate-<pid>` and `CARGO_BUILD_JOBS=2` so background
  local gates do not drift into ad hoc artifact roots or unbounded build
  parallelism.
- `tools/ci_gate.py` runs each child check under `tools/memory_guard.py` by
  default. The guard enforces per-process and process-tree RSS ceilings. Default
  budgets adapt to live available memory and physical RAM, leaving a reserve so
  workstation sweeps can use large-memory machines without risking system-wide
  memory pressure; the reserve floor stays small on hosted runners so compiler
  children are not forced into allocator aborts before the guard can report RSS
  evidence. The inherited direct-child OS resource limit is a separate adaptive
  virtual-memory clamp (`--child-rlimit-gb`) so Node/WASM-style address-space
  reservations do not get reported as real RSS pressure. GitHub smoke workflows
  wrap direct pytest and Cargo invocations with the same guard instead of relying
  only on nested harness-level guards.
  Standalone guarded commands support
  explicit wall-clock timeouts via `--timeout` and per-poll RSS telemetry via
  `--samples-jsonl`; summaries also classify signal-style child exits such as
  `143`/`SIGTERM` when no guard RSS or timeout violation occurred.
  `--no-memory-guard` is a diagnostic escape hatch, not a normal CI mode.
- The `tools/memory_guard.py` CLI re-execs an internal worker before launching
  the guarded command, passing the child command through an internal JSON
  environment variable instead of the guard process argv. This keeps
  process-name tools such as `pkill -f <child-pattern>` from matching and
  killing the guard parent while still stripping the internal environment keys
  before the child command starts.
- Differential sweeps also run an in-harness recursive RSS guard. It writes
  only bounded telemetry under `MOLT_DIFF_ROOT/memory_guard/`: `tripped.json`,
  `events.jsonl`, `global_samples.jsonl`, and at most one rotated sibling for
  each stream. The active subprocess and cumulative monitor refresh adaptive
  caps on every poll and account active Molt tree RSS back into the budget, so
  the guard scales up on large-memory machines and scales down under unrelated
  workload pressure without self-tightening on memory Molt is already using.
  `tools/memory_guard_stream.py` is the live viewer; it tails those compact
  NDJSON files and does not add another process sampler or artifact tree.
- Operators can avoid sample artifacts entirely with
  `MOLT_DIFF_MEMORY_GUARD_WRITE_SAMPLES=0` and stream samples directly from the
  harness via `MOLT_DIFF_MEMORY_GUARD_STREAM=stderr|stdout|json-stderr|json-stdout`.
  Trip/event evidence remains bounded and canonical.
- Standalone guarded commands use the same stream vocabulary through
  `tools/memory_guard.py --stream stderr|stdout|json-stderr|json-stdout`.
  `--samples-jsonl` is bounded by default and rotates one sibling via
  `--samples-max-mb` so ad hoc local runs do not create unbounded telemetry.
- Standalone and CI-gate guarded commands apply a direct-child OS resource limit
  before `exec` (`--child-rlimit-gb`, defaulting to the adaptive child
  virtual-memory clamp) so fast allocators have a kernel-backed ceiling in
  addition to recursive process-tree polling.
- WASM CI runs linked Node/V8 parity with a 128 GB child virtual-address-space
  floor. This prevents V8 WebAssembly reservations from surfacing as host
  SIGSEGVs while the default-on RSS/tree/global guard remains the source of
  real memory-pressure enforcement.
- Guard-wrapped CI commands that need the synced project Python environment run
  through `tools/venv_exec.py` after `uv sync` instead of wrapping the guarded
  command in `uv run`. The shim activates the repository virtualenv with
  `VIRTUAL_ENV` and `PATH` and then `exec`s the requested command, so
  `tools/guarded_exec.py` owns the real child process tree and can classify or
  clean descendants without a lingering package-manager process layer.
- WASM CI disables the persistent backend daemon with `MOLT_BACKEND_DAEMON=0`
  because each GitHub step is its own standalone guarded process tree. Warm
  backend daemons remain the developer and benchmark default where a
  suite-level sentinel can own the daemon lifecycle; these one-shot CI compile
  steps must finish with no live helper descendants, stale daemon sockets, or
  guard-side orphan cleanup.
- `tools/ci_gate.py --parallel` must clamp worker count by the active memory
  guard budget when enabled. The default cumulative cap is adaptive from live
  available memory and physical RAM, with a reserve for OS and non-Molt
  workload pressure. Differential auto-parallelism uses a scheduler budget
  separate from the per-tree kill ceiling so a large-memory host is not
  serialized by a worst-case single-tree cap.
- `tools/process_sentinel.py` is the canonical out-of-band cleanup and live
  sentinel for repo-scoped Molt build/test/bench process groups that were not
  launched by the current guard. It must stream by default and avoid artifact
  writes unless a future workflow adds an explicit bounded output. Use
  `--until-clean-sec` with `--max-runtime-sec` for bounded drains of delayed
  stale launches; avoid shell loops for this cleanup pattern. It must enforce
  per-process, per-process-group, and all-matched-groups aggregate RSS ceilings
  with the shared adaptive live-memory defaults unless explicit limits are set.

## 2. Jobs

### 2.1 Push-Time Presubmit (`ci.yml`)

- `docs-gates`
  - generated doc freshness
  - docs architecture contract
  - focused docs tests
- `python-tooling-smoke`
  - cheap Python/tooling smoke tests and import checks
- `rust-build-unit-smoke`
  - Linux build plus selected Rust unit-smoke crates

### 2.2 Nightly Correctness (`nightly.yml`)

- `molt-conformance-full`
  - canonical `run_molt_conformance.py --suite full`
- `differential-basic-stdlib`
  - curated `tests/differential/basic` and `tests/differential/stdlib`
- existing periodic heavy lanes such as regrtest and determinism sweeps remain here

### 2.3 Release Validation (`release.yml`)

- explicit hosted release validation across supported macOS and Linux lanes
- artifact upload/publication checks
- release manifest generation

### 2.4 Perf Validation (`perf-validation.yml`)

- manual benchmark execution
- artifact upload for result inspection

## 3. Caching Strategy (recommended)
- **Python**: Cache `~/.cache/uv`.
- **Rust**: Cache `target/` and `~/.cargo/registry` using `swatinem/rust-cache`.
- **WASM**: Cache pre-compiled Molt Packages when WASM packages land.

## 4. Release Trigger
- Pushing a release tag (or manually dispatching `release.yml`) triggers hosted release validation and publication.
