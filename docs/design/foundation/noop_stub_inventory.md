## P0 — silent-wrong-answer / permanent-leak (fix first)

### `molt_gc_collect` — `runtime/molt-runtime/src/object/ops.rs:1810`

**Why P0:** VERIFIED at source. gc.collect() does NO object-graph traversal and NO cycle reclamation: its whole body is `weakref_collect_for_gc(_py)` (which only clears weakrefs to ALREADY-dead targets, frees nothing live) + `state.count=(0,0,0)` + returns the weakref-clear count as the collected count. Confirmed there is NO cyclic collector anywhere in the tree (zero hits for trial_deletion/tarjan/tp_traverse/mark_and_sweep). Consequence: every reference cycle (a.b=b;b.a=a; a list containing itself; obj.__dict__ back-ref) leaks FOREVER under molt while gc.collect() reports success with a fabricated count. This is a permanent memory-model divergence from CPython and a leak class the council's MOLT_ASSERT_NO_LEAK rule cannot even catch (it counts RC destruction, not cycle reclamation). Corollary dead gauges: GcState.count is never fed by allocation so gc.get_count() is inert; gc.get_threshold() returns (0,0,0) instead of CPython's (700,10,10).

**Fix:** Build a real cyclic collector on the council's ownership-lattice arc: (1) maintain a tracked-set of GC-eligible container objects (list/dict/set/tuple-of-containers/instances-with-__dict__) registered at allocation and unregistered at free; (2) implement CPython's subtract-internal-references / trial-deletion (gc_refs) over that set to find objects unreachable except through cycles; (3) run their finalizers (__del__/tp_finalize) under the ownership-lattice ordering with CPython resurrection semantics (reusing the existing faithful maybe_run_object_finalizer); (4) break cycles, reclaim memory, populate gc.garbage with uncollectable instances; (5) return the TRUE count. Drive cadence from real generations/thresholds seeded to (700,10,10). If a no-cycle-collector design is ever a deliberate product decision, gc.collect() must NOT claim success — it must document/raise the RC-only limitation, never report a fabricated count. This is the headline P0 of the entire audit.

### `FractionHandle::hash_val (molt_fraction_hash)` — `runtime/molt-runtime/src/builtins/fractions.rs:116`

**Why P0:** VERIFIED at source (and to_i64 returns None, not a truncation, confirmed at runtime-math/src/bridge.rs:289). For a whole-number Fraction it returns `numer.to_i64().unwrap_or(0)` — so EVERY Fraction whose numerator exceeds i64 (e.g. Fraction(10**30)) hashes to 0, colliding en masse. For a non-whole Fraction it returns `f.to_bits() as i64` (the raw IEEE-754 bit pattern of the f64 approximation), e.g. hash(Fraction(1,2)) = 0x3FE0000000000000 instead of CPython's hash(0.5)=1152921504606846976. This BREAKS the numeric-hash invariant: a Fraction no longer hashes equal to a numerically-equal int/float/Decimal, so dict/set membership across numeric types is silently wrong (Fraction(1,2) and 0.5 land in different buckets; {Fraction(1,2):'x'}[0.5] raises KeyError where CPython succeeds). Silent-wrong-answer in a core container invariant = P0.

**Fix:** Implement CPython's exact numeric _hash_algorithm: _PyHASH_MODULUS = 2**61-1 (on 64-bit); dinv = pow(denominator, -1, _PyHASH_MODULUS) via modular inverse; if denominator is divisible by the modulus use _PyHASH_INF; hash_ = hash((hash(abs(numerator)) * dinv) % _PyHASH_MODULUS); result = hash_ if numerator>=0 else -hash_; map -1 -> -2. hash(abs(numerator)) must itself be the CPython BIGINT hash (n mod _PyHASH_MODULUS, sign-applied), NOT an i64 cast. Route int-hash through the existing BigInt path so large whole Fractions hash correctly and equal their int. This is the same modular-hash authority Decimal/float/int already need — unify it.

### `parse_ipv6_from_bits / molt_ipaddress_v6_new` — `runtime/molt-runtime/src/builtins/ipaddress.rs:223 (parse_ipv6_from_bits) via molt_ipaddress_v6_new:379`

**Why P0:** VERIFIED at source. IPv6Address(int) reads the int via to_i64 (i64 range) then casts to u128; since to_i64 returns None for ints > i64 (confirmed bridge.rs:289), every IPv6 integer >= 2**63 raises TypeError 'expected str or int for IPv6 address'. The ENTIRE UPPER HALF of the IPv6 address space is un-constructable from an int — IPv6Address(2**128-1) (the valid all-ones address) fails where CPython succeeds. The Python wrapper passes the int straight through with no compensation. This is a hard parity break that makes a large class of valid programs (anything constructing real-world global-unicast / link-local IPv6 from packed ints) crash.

**Fix:** Read the int as a full u128 via the BigInt path (to_bigint), range-check 0 <= n < 2**128, and convert BigInt -> [u8;16] big-endian (matching the IPv4 path's to-bytes logic widened to 16 bytes). Raise AddressValueError-equivalent only for negative or >=2**128 values, exactly as CPython accepts every 0..2**128-1 integer. Audit molt_ipaddress_v4_new for the same to_i64 ceiling (IPv4 fits in i64 so it is safe, but confirm) and any network/interface constructors that take ints.

### `transcendental_via_f64 / Decimal.sqrt/ln/log10/exp/pow(non-integer)` — `runtime/molt-runtime/src/builtins/decimal_without_mpdec.rs:1319 (transcendental_via_f64) consumed by molt_decimal_sqrt:1675, molt_decimal_ln:1704, molt_decimal_log10:1733, molt_decimal_exp:1192, molt_decimal_pow:1581`

**Why P0:** VERIFIED at source. The body does `text.parse::<f64>()` -> `f(base)` (libm double) -> `format!("{:.*e}", prec+4, val)` -> reparse. f64 carries only ~15-16 significant decimal digits, so EVERY digit beyond ~16 of a >=28-digit context-precision result is garbage from the binary->decimal round trip: Decimal(2).sqrt() at default prec=28 must be 1.414213562373095048801688724 but yields ~16 correct digits + noise. It also unconditionally stamps MPD_INEXACT|MPD_ROUNDED even when CPython is exact (Decimal(4).sqrt()==2). This is an approximation passed off as the real arbitrary-precision semantics. CRITICAL: this path is LIVE on every build WITHOUT bundled libmpdec — notably the entire wasm32 target and any host where build_libmpdec fails (decimal.rs selects decimal_without_mpdec when cfg(molt_has_mpdec) is unset). So a major target ships silently-wrong Decimal math.

**Fix:** Implement true arbitrary-precision correctly-rounded transcendentals on the BigInt coefficient (as libmpdec/_decimal does): integer Newton iteration for sqrt to prec digits with exact-result detection and correct final rounding; Taylor/AGM exp/ln/log10 at prec+guard digits with correct rounding and proper Inexact/Rounded/Underflow signalling; pow via exp(b*ln(a)) with the spec special-case ladder. NEVER route Decimal arithmetic through f64. Until landed, the honest interim is to make these raise/refuse on the no-mpdec path rather than return wrong digits — but the structural fix is the native big-coefficient implementation so wasm32 has real Decimal.

## Systemic patterns (fix the structural cause, not each instance)

- CONTEXT-DEFAULT THEATER in decimal_without_mpdec.rs: a whole family of methods (is_normal/is_subnormal/number_class, next_plus/next_minus) invent a phantom Emin = 1 - prec instead of reading the context's real Emin/Emax (defaults -999999/+999999), which the DecimalContextHandle does not even STORE. This is one structural cause (missing Emin/Emax fields on the context) producing 4+ distinct parity breaks. Root-fix: add Emin/Emax to the context and derive Etiny/Nmax from them everywhere.
- NUMERIC-HASH DIVERGENCE: Fraction.hash_val returns f64 bit-patterns / i64-truncated values instead of CPython's modular _hash_algorithm. The same modular-hash authority (mod 2**61-1, sign-mapped, -1->-2) is needed by Fraction, Decimal, big int, and float for the cross-type numeric-hash invariant. Today it is scattered/missing. Structural cause: no single shared py_numeric_hash primitive. (datetime.rs DOES port CPython's xxHash tuplehash correctly — confirmed at source — so the authority exists for tuples but not for rationals.)
- DUPLICATE-AUTHORITY PARITY BREAK: is_printable_for_repr (ops_format.rs) hand-rolls a Unicode-printability approximation that disagrees with the GENERATED unicode_printable_table::is_printable that str.isprintable() already uses — so repr() and isprintable() give different answers for the same Cf/Zs/Co/Cn code points. Recurring pattern: a second hand-rolled copy of a fact that already has a generated single-source authority. Fix = route the second consumer through the generated table.
- WASM SUCCESS-THEATER VIA MACRO: the wasm_net_stub! macro (sockets.rs:6684) mass-generates ~30 socket fns whose entire body is `MoltObject::none().bits()` — returning None as if the op succeeded. This conflates THREE different correct behaviors into one wrong one: (a) pure CPU computations that should run identically on WASM (htons/ntohs/htonl/ntohl/inet_pton/inet_ntop/cmsg_len/cmsg_space — VERIFIED at source, native impls are pure one-liners), (b) host-dependent ops that should RAISE OSError like molt's own native fallbacks (getaddrinfo/gethostname/sendfile/sethostname), and (c) ops that already have a WASM host-import path (getaddrinfo_host). Structural cause: a blanket 'return None' macro instead of three honest categories (compute / raise-unsupported / host-import).
- RESOLVED 2026-06-26: wasm_net_stub! was deleted. WASM now compiles `sockets_net.rs` as the canonical socket utility authority, so pure helpers run there, host-import helpers route through the existing WASM hooks, unsupported helpers raise explicit errors, and no socket utility export is synthesized by a blanket None-returning macro.
- ANNOTATION-ONLY OPTIMIZER THEATER in the TIR pass pipeline: vectorize.rs and polyhedral.rs run on every function in build_default_pipeline, claim a named optimization (SIMD vectorization / polyhedral tiling), set attrs (vectorize/reduction/simd_width/promoted; polyhedral_tileable/tile_size), and those attrs are read by NOTHING — the LLVM backend reads `vectorize` as `let _ = has_attr(...)` at 5 sites and discards it; no LLVMSetMetadata/llvm.loop.vectorize is ever emitted. Burns compile time and falsely advertises optimizations. (fast_math.rs is the correct counter-pattern: writes an attr the backend genuinely consumes via set_fast_math_flags.) Fix: either wire the attr->codegen (raw llvm-sys LLVMSetMetadata) or delete the pass from the pipeline.
- STORED-BUT-NEVER-FIRED HOOK THEATER: sys.settrace/sys.setprofile validate+store the callable and return success, but trace_profile/trace_bits/profile_bits are consumed ONLY by their own getters — no compiled-code instrumentation ever fires call/line/return/exception events (the project's own sys.py:1443 admits 'Molt does not support tracing'). Same shape as molt_memoryview_release (stores nothing, models no released state) and gil_check (computes is_gil_released() then does nothing). Pattern: an entry point that records intent but never actuates it, reporting success the whole time.
- DEAD-BUT-DANGEROUS SCAFFOLDING that misrepresents itself by name/doc and would be a P0 if revived: the entire deopt.rs subsystem (advertises 32-value live capture + transfer, but no pass emits OpCode::Deopt, the registry is permanently empty, it forwards only values[0], and the LLVM call site passes an i64 where the Rust ABI expects *const DeoptState — memory-unsafe if it ever fired); dict_compact.rs::delete (swap-with-last, silently violating CPython insertion-order, but nothing constructs CompactDict); molt_list_int_setitem_raw (documents/implements no-op-on-OOB, diverging from IndexError, registered in the WASM import table but emitted by no backend). Structural cause: half-wired subsystems left in the tree advertising contracts they don't deliver and inviting a future caller to reintroduce the divergence.
