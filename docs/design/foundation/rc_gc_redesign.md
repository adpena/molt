## Recommended architecture

molt adopts a TWO-TIER hybrid that is exactly CPython's memory model, adapted to molt's NaN-boxed AOT runtime and held to a Perceus-class precision/perf bar:

TIER 1 — PRECISE COMPILE-TIME REFERENCE COUNTING (Perceus-class), the primary reclaimer for the acyclic majority. This is molt's existing TIR drop-insertion pipeline (drop_insertion.rs + liveness.rs + alias_analysis.rs + refcount_elim.rs + ownership_lattice_min.rs), KEPT and COMPLETED — not replaced. It is already a faithful Perceus elaboration over MLIR-block-args-as-phis with a borrow-args (+0 operand / +1 result) ABI = Perceus's Delta(borrowed)/Gamma(owned) split. The single structural defect is that the Perceus mutable-slot rule `(:=) = read-old; overwrite; drop-old` is not discharged on every Python slot-overwrite shape: §2.7 loop-carried is a documented NO-OP (drop_insertion.rs:2992-3002), and the molt_main / module-dict and boxed/async-closure lanes are excluded from the named-local lifetime machinery (local_bindings.py gates on `current_func_name != "molt_main"`). Tier 1 closes this by making "a slot/dict store of an owned value is a drop obligation discharged exactly once on every path including back-edges" a first-class invariant on the minimal ownership lattice, verified by a consume-exactly-once translation validator (molt-check) so the bug class becomes unexpressible.

TIER 2 — A CPYTHON-PARITY CYCLE COLLECTOR, scoped to gc-trackable container types, run only on threshold like CPython, never touching the acyclic fast path. It replaces the molt_gc_collect stub (ops.rs:1810) with a faithful re-implementation of CPython 3.12 gc_collect_main's partition algorithm (deduce_unreachable = update_refs/subtract_refs/move_unreachable) and exact 6-step destruction order, REUSING molt's existing finalizer (maybe_run_object_finalizer) and weakref (weakref_clear_for_ptr) machinery so cyclic and acyclic finalization are byte-identical. Data structures adapt to molt's perf contract: a side registry of tracked CONTAINS_REFS containers (NOT a per-object 16-byte PyGC_Head on the hot 24-byte header), the 2-bit GC color + buffered + tracked bits packed into the FREE bits 24-31 of the existing flags word, and transient gc_refs scratch (cold-header slab or a per-collection map) since collection is stop-the-world under the GIL.

THE SHARED KEYSTONE that both tiers and all four backends require: ONE canonical, generated `molt_traverse(ptr, &mut dyn FnMut(child_bits))` (a real tp_traverse) plus a `molt_clear(ptr)` (tp_clear), extracted from the per-type child-enumeration currently inlined in dec_ref_ptr's dealloc cascade (mod.rs:2188+), with a CI drift gate asserting traverse-set == dealloc-walk-set per type_id. This single authority feeds the deallocator, the cycle collector's mark/clear, and gc.get_referents/get_objects parity.

THE SHARED FACT PLANE (council "one cached ClassInfo fact" doctrine): two sibling cached, MRO/version-derived facts drive everything — FinalizerSensitive (already exists) and a new MayFormCycle (the Bacon-Rajan GREEN/acyclic bit), computed sound-conservatively (GREEN only for int/float/bool/str/bytes/None and tuples transitively of those; default cycle-capable). MayFormCycle is consumed by (a) the allocator (register in the tracked set only if non-GREEN), (b) the drop pass (no candidate-registration emit for GREEN roots), and (c) escape/refcount-elim/stack-alloc/Free-eligibility. This makes the acyclic majority pay ZERO collector cost — the property that lets molt beat both CPython (which tracks all containers unconditionally) and Codon (which leaks cycles).

RC stays NON-MOVING (no stack maps, no write barriers, no safepoints — backend-uniform inc/dec), which is decisive given four backends including WASM (no precise stack scan) and Luau (host GC). Tracing/MMTk are explicitly rejected: they would break __del__ promptness/ordering parity and impose a 4× per-backend stack-map tax with a WASM hole, for no perf gain (Down-for-the-Count/RC-Immix prove optimized RC ties/beats the best tracer).

## Tier 1 — Perceus-class precise compile-time RC (fixes the 513 MB acyclic leak class)

PERCEUS-CLASS PRECISE COMPILE-TIME RC — the design that provably cannot miss the implicit-drop class molt currently leaks.

FORMAL NORTH STAR (Perceus, MSR-TR-2020-42 v4 / PLDI'21): the syntax-directed judgement `Delta | Gamma |-s e ~> e'` with Delta=borrowed, Gamma=owned, four by-construction invariants (Delta∩Gamma=∅; Gamma⊆fv(e); fv(e)⊆Delta,Gamma; multiplicity exactly 1). molt's pass already implements the SSA analogue: liveness.rs = owned/free-variable tracking (alias-root space, representation-filtered so raw i64/bool/f64/None carry no RC); alias_analysis.rs = union-find folding Copy/TypeGuard chains to one owned root; drop_insertion.rs §1 = consume-at-last-use ([svar]), §1b = unused-binding drop ([sbind-d]), §3 edge-dying = per-branch `drop Gamma'_i` ([smatch]), §5 mixed-ownership-phi IncRef = borrow-into-phi `dup` ([svar-dup]); the +0 operand/+1 result ABI = Delta/Gamma parameter split; ownership_lattice_min.rs FinalizerSensitive = the necessary divergence for CPython __del__ ordering (no Perceus analogue). KEEP all of this.

THE ROOT-CAUSE FIX (the load-bearing invariant). The leaking program `for i in range(1e6): d = {...}` is a Python MUTABLE-SLOT OVERWRITE whose Perceus shape is `(:=) ≡ val y = slot; slot := new; drop y` (TR §2.7.3). molt fails to emit `drop y` because the slot value never becomes a visible OWNED resource the rules can see, across THREE non-SSA-phi lanes:
  (1) §2.7 loop-carried is a NO-OP. For a function-local loop-carried OWNED block-arg phi `p` that is DEAD on the back-edge (overwritten without being read that iteration), §1 (operand last-use) misses it (a pure-reassign body never reads p), §1b (op.results only) misses it (p is a block arg not a result), §3 edge-dying EXPLICITLY skips block args (drop_insertion.rs:2921), §5 only adds IncRefs. So only the FINAL value is dropped at loop exit; the N-1 intermediates leak (measured: dict alloc=1000098 / dealloc=2 / live=1000096 inside `def go()`).
  (2) molt_main / module-dict lane: top-level loop vars route through MODULE_SET_ATTR; the runtime dict_set_in_place (ops.rs:9529-9542) correctly inc-new/dec-old, so the LEAK there is the BuildDict PRODUCER temp pinned because its §1 last-use DecRef after the store is suppressed (the temp appears live-out via self.globals/self.locals caching).
  (3) boxed-cell / async-closure locals similarly bypass _plain_local_del_boundary_value.

STRUCTURAL FIX (one rule, whole class, on the ownership lattice — rung-3 "ordered release obligation", NOT a placement special-case):
  A. Implement §2.7 as specified: for each back-edge `→ header(args)`, for each arg position p binding an OWNED droppable header phi `header.args[p]`, if the phi is DEAD at the back-edge (not read between header entry and the latch's computation of the new value), insert `DecRef(header.args[p])` immediately before the back-edge branch. Gate strictly on dead-at-back-edge so the accumulator shape (`total = total + x`, which reads the phi and is already dropped by §1 at that read) is NOT double-dropped. Path-condition it for conditional rebinds (`if c: d = ...`): only release on paths that overwrote the slot.
  B. Unify the molt_main / module-dict lane into the SAME TIR overwrite-release obligation: either thread module-global stores through the bound_local boundary machinery or kill the self.globals/self.locals phantom-live-out so §1 sees the producer last-used at MODULE_SET_ATTR. Retire the legacy backend `loop_body_out_vars` name-based dec_ref lane (function_compiler.rs:232-271) ATOMICALLY in the same change (it is disabled when drop_inserted, so two lanes coexist — a double-drop hazard if both ever fire). REJECT re-enabling the legacy lane or special-casing module scope (recreates the two-source-of-truth trap).
  C. Extend the obligation to finalizer-sensitive locals with CPython-correct prompt-at-rebind __del__ timing (the old occupant's finalizer runs when the slot is rebound), consulting FinalizerSensitive so the release point matches CPython observable ordering.
  D. ACTIVATION COMPLETENESS (largest blast radius, do this on the same lattice): today the pass BAILS entirely (zero ordinary-temp drops) for any function with try/except or any generator/async body (drop_insertion.rs:1127 has_exception_handlers/has_state_machine), so every heap temp in real Python code leaks. Replace the bail with op-granular dominance over the exception-augmented + state-dispatch CFG so drops place cross-block without the use-before-def the bail exists to prevent. This must land FAIL-CLOSED-preserving (the current bail leaks but never UAFs; the completion must not introduce double-free) and with the full leak/finalizer/weakref/unwind test battery green.

VERIFICATION-AS-INVARIANT (Swift OSSA SILVerifier analogue): build the molt-check translation validator that proves every OWNED root has EXACTLY ONE release on EVERY path (including back-edges and exception edges). An owned root with zero releases on a path is a verifier error — this makes the entire missing-implicit-drop class unexpressible at the IR level and would have caught the current leak. Repr only moving UP the lattice is the companion drift gate.

PERF FRONTIER (after correctness, in order; each compounds with the raw-scalar keystone since a value in a raw i64/f64 lane carries NO refcount): (1) Nim cursor inference — `dest = src` where src is param-derived and unmutated on all paths becomes a non-owning alias (no inc, no drop), extending alias_analysis.rs; fail-closed to an owned copy if non-mutation cannot be proven on all paths (UAF hazard under NaN-boxing — Miri-strict-provenance must cover the cursor). (2) Swift RC-Identity — codify `~rci` over Copy/TypeGuard/payload-extract so an IncRef on one handle cancels a DecRef on an RC-identical handle, including pairing iteration-N release with iteration-N+1 retain, extending refcount_elim.rs. (3) Perceus drop-specialization (`if is_unique then free-children;free else decref`) + drop-guided FRAME-LIMITED reuse (Lorenzen/Leijen ICFP'22, performed AFTER rc insertion, safe-for-space) so the dict-in-a-loop becomes in-place mutation. (4) Non-atomic RC under the GIL + re-enabled inlined RC codegen (the per-op cost lane — see perf_model).

## Tier 2 — CPython-parity cycle collector

CPYTHON-PARITY CYCLE COLLECTOR — faithful re-implementation of CPython 3.12 gc_collect_main, adapted to molt's runtime.

ALGORITHM (CPython's partition form, which is the proven DUAL of Bacon-Rajan trial deletion per the Unified Theory — same garbage set, but iterative/no-recursion and bit-exact gc-module parity for free):
  deduce_unreachable(generation):
    update_refs(): snapshot each tracked object's ref_count into a transient gc_refs scratch (cold-header slab field via the existing cold_idx, or a per-collection HashMap<PtrSlot,isize>); mark COLLECTING.
    subtract_refs(): for each tracked object call molt_traverse; the visitor decrements gc_refs of each referent that is itself in the candidate set. After this, gc_refs>0 ⟺ referenced from OUTSIDE the set (a root); gc_refs==0 ⟺ cycle candidate.
    move_unreachable(): iterative BFS — gc_refs>0 objects are roots and re-mark transitive referents reachable (pulled-back objects get gc_refs:=1); the remaining gc_refs==0 objects move to the `unreachable` list (NEXT_MASK_UNREACHABLE). Survivors end up reversed so later scans never move them.

DESTRUCTION ORDER (verbatim from CPython 3.12 gc_collect_main — the single most parity-sensitive contract; do NOT reorder):
  1. handle_weakrefs(unreachable): FIRST clear all weakrefs pointing INTO the unreachable set (so they read None / never expose a dying object); enqueue callbacks ONLY for weakrefs that are themselves reachable (a weakref unreachable in the same cycle has its callback DROPPED); THEN invoke those callbacks. Reuse weakref_clear_for_ptr / the runtime_state.weakrefs side-table. Weakref clearing STRICTLY PRECEDES finalizers.
  2. move_legacy_finalizers: molt has NO legacy tp_del (all finalizers are tp_finalize-class __del__), so this collapses and gc.garbage is ALWAYS empty (matching modern CPython post-PEP-442 where every __del__-bearing cycle IS collectable). Keep gc.garbage as an always-empty list for API parity.
  3. finalize_garbage(unreachable): for each object with HEADER_FLAG_INSTANCE_HAS_FINALIZER/CLASS_HAS_FINALIZER and not HEADER_FLAG_FINALIZER_RAN, set FINALIZER_RAN (PEP 442 run-once) and call THE EXISTING maybe_run_object_finalizer (mod.rs:1810-1968) — NOT a second finalizer path — in deterministic GC-list (insertion) order, never address/hash order.
  4. handle_resurrected_objects: RE-RUN deduce_unreachable on the post-finalization set; anything a __del__ resurrected (re-incremented / re-linked to a live root) leaves the collectable set and merges into the old generation. This is the SET-level analogue of the existing single-object resurrection abort and is mandatory (omitting it = use-after-free on resurrected objects — the council P0 corruption surface).
  5. clear any weakrefs created during finalizers.
  6. delete_garbage(final_unreachable): call molt_clear (tp_clear) on each still-unreachable object — decref + empty its backing store in place WITHOUT freeing the container — so the RC cascade collapses the cycle. molt_clear MUST NOT free (freeing while on the unreachable list double-frees).
  gc.collect() returns n+m (n=uncollectable=0 in molt, m=collected).

DATA STRUCTURES (adapted to molt's perf contract — REJECT a per-object 16-byte PyGC_Head on the hot 24-byte header):
  - Tracked-object side registry: a generational structure (3 doubly-linked intrusive lists or a generational set keyed by PtrSlot via expose_provenance/resolve_ptr — never raw `as usize`, for Miri strict-provenance) of CONTAINS_REFS containers, populated at container alloc, removed on free. GREEN/atomic types are never registered.
  - Color + bookkeeping bits: 2-bit GC color (black=0/gray/white/purple) + 1-bit buffered + 1-bit tracked packed into the FREE bits 24-31 of the existing flags:u32 word (confirmed free) — zero header growth, colors live on-heap not in the NaN-boxed value (strict-provenance preserved).
  - gc_refs scratch: transient, collection-time-only (stop-the-world under the GIL makes a permanent field unnecessary).
  - Generations + thresholds: wire CPython 3.12+ defaults (2000,10,10) — NOT the stale (700,10,10) — onto the existing gc_state {count,thresholds,enabled,debug_flags}; tracked-container allocs bump count0; cross-threshold triggers gen0, with gen1/gen2 escalation and the long_lived_pending < long_lived_total/4 (25%) full-collection guard.
  - Untracking parity: replicate untrack_tuples/untrack_dicts by clearing CONTAINS_REFS on containers proven atomic-only, so gc.is_tracked((1,2,3)) == False exactly as CPython.

API SURFACE (exact contracts): collect(generation=2)->n+m; get_count/get_threshold/set_threshold (default 2000,10,10); get_objects(generation)/get_referrers/get_referents (walk molt_traverse + registry); is_tracked/is_finalized; get_stats (per-generation collections/collected/uncollectable); enable/disable/isenabled; freeze/unfreeze/get_freeze_count (permanent generation); DEBUG_* flags; gc.callbacks(phase∈{start,stop}, info). gc.garbage always empty.

FORMAL: extend formal/quint/molt_gc_safety.qnt (currently proves leak-freedom for ACYCLIC graphs only — its `collect` action just cascades dangling-edge decrefs) with a trial-deletion `cycleCollect` action and an "unreachable cycle is eventually reclaimed" invariant. Acceptance gates: tests/differential/stdlib/gc_cycle_collector.py (weakref.ref(first)() is None after gc.collect()) and tests/differential/memory/cycle_leak_negative.py (the falsification artifact, flipped to a positive assertion).

BACKEND PARITY: the collector is runtime code (molt-runtime), backend-agnostic (walks heap objects, not generated code). Registration is emitted via a portable intrinsic only for non-GREEN roots. On Luau (host GC) molt objects must be reclaimed by molt's collector, not double-managed; tracked-registry state is per-runtime-instance, not process-global. ## Performance model

PERFORMANCE MODEL — beat CPython on every benchmark, approach Codon (AOT) and PyPy (dynamic), across all four backends.

TWO REGIMES, attributed per benchmark (never conflated):
  NUMERIC/LOOP KERNELS → the raw-scalar lane carries NO refcount (drop_insertion filters RawI64Safe/Bool/FloatUnboxed out of liveness), so every value the repr lattice hoists out of NaN-boxed form is an inc/dec DELETED. Combined with CheckedMul peel / RawI64Safe accumulators this is the Codon-class path (10-100×+ CPython). RC perf and numeric-repr perf are the SAME compression ladder: widen the raw lane → fewer ops; cheapen the residual boxed ops → faster ops.
  BOXED/DYNAMIC LOOPS → the per-op cost of surviving RC dominates.

PER-OP COST MODEL (native, GIL held, one surviving boxed op):
  today    ≈ out-of-line C-ABI call (spill + GIL-TLS re-entry) + atomic RMW           ≈ 5-15 ns  (LOSES to CPython's inlined non-atomic Py_INCREF)
  +nonatomic (Phase 7) ≈ call + non-atomic load-add-store                              ≈ 4-8 ns   (~5× cheaper memory op; ~10ns→~2ns; larger win on ARM where AcqRel ldaxr/stlxr → plain add)
  +inlined (Phase 7)   inc ≈ load_flags; select_delta; non-atomic add                  ≈ ~2 ns    (BELOW CPython — AOT constant-folds the tag/immortal tests CPython runs at runtime)
                       dec ≈ load_flags; -1; brif rc==0 → out-of-line slow free        ≈ ~2 ns hot
  +borrowed/cursor     ≈ 0 (no op emitted)
  +immortal (None/True/small-int/interned/type/code) ≈ 0 op AND 0 cache-line invalidation
  +raw-lane            ≈ 0 (value carries no refcount)

CYCLE COLLECTOR COST: kappa = O(N_cand + E_cand) over the TRACKED candidate set per collection (NOT whole heap; survivors reversed once then never moved — CPython's amortization). phi (frequency) tracks CPython via the (2000,10,10) thresholds + 25% guard. Mutator overhead with threshold-triggered (not per-decrement-buffered) tracking: ZERO added branches on the RC dec hot path; only two pointer writes (tracked-list link/unlink) + a counter bump on non-GREEN container alloc/free. GREEN/atomic objects pay literally nothing (no header growth, no list, no scan).

NET vs REFERENCES:
  vs CPython (floor, must beat on EVERY benchmark/backend/profile): molt does strictly LESS cyclic-GC work (AOT proves acyclicity → more GREEN/untracked objects, smaller candidate sets), reclaims the acyclic majority by RC with zero GC involvement, and after Phase 7 has cheaper-than-CPython residual RC ops. The 513MB leak fix is pure win (O(n)→O(1) peak RSS, allocator/page-in pressure removed).
  vs Codon (AOT north star): numeric/typed kernels approach/exceed via raw-lane + drop-specialization + frame-limited reuse; molt additionally has CPython-correct cycles + finalizers (Codon leaks cycles via Boehm) — a correctness win, reported as non-equivalent semantics not a speed loss.
  vs PyPy (dynamic reference): boxed/dynamic loops match/beat via inlined non-atomic RC + IC tiering; where PyPy wins, NAME the missing fact (loop specialization / shape propagation) rather than hiding it.

METHODOLOGY (release-blocking, pyperf discipline): every claim reports benchmark→target→backend→profile→CPython ratio→PyPy ratio→Codon ratio→binary size→peak RSS→compile time→cold AND warm→artifact; quiescent, repeated, attributed; classified GREEN/RED_STABLE/RED_NOISY/TIE/DIMENSIONAL_WIN. A reuse/alloc/RSS-only improvement is reported as DIMENSIONAL_WIN, never as a warm-speed heal. Separate scoreboards for x86 AND ARM (the non-atomic win is larger on ARM). No perf optimization on a still-leaking path.

## Provenance & safety (Miri strict-provenance)

MIRI STRICT-PROVENANCE CORRECTNESS for the NaN-boxed unsafe core.

THREAT MODEL: NaN-boxing packs heap pointers into f64 bit patterns; any pointer-as-integer round-trip that uses raw `ptr as usize` / `usize as ptr` loses provenance and fails Miri strict-provenance (and is UB under the Rust provenance model). The cycle collector's registry and gc_refs map, plus the new cursor/reuse machinery, all touch raw pointers and are new exposure surface.

RULES (binding for all new RC/GC code):
  1. ALL pointer↔integer round-trips go through the established expose_provenance/resolve_ptr path (already used by weakref.rs:69), NEVER raw `as usize`. The tracked-object registry and the transient gc_refs scratch key on PtrSlot via this path. PtrSlot already carries the raw pointer with the right Send/Sync discipline; extend it to round-trip through with_exposed_provenance, not integer casts.
  2. The color/buffered/tracked bits live in the on-heap MoltHeader.flags word (bits 24-31), NEVER in the NaN-boxed value — so boxing/unboxing never sees GC state and provenance of the boxed pointer is untouched.
  3. molt_traverse/molt_clear receive child references as the runtime's MoltObject bits and resolve them through the existing typed accessors (seq_vec_ptr, dict order/table, fields_ptr) which already carry provenance — the visitor must pass bits, and the collector must resolve via the same accessors, not reconstruct pointers from integers.
  4. Cursor inference (Phase 7): a cursor is a non-owning alias of a borrowed location. Under NaN-boxing a stale cursor is a dangling pointer inside an f64; cursors must be proven non-mutated/non-freed on ALL paths between creation and last use (fail-closed to an owned copy otherwise), and the cursor value must retain provenance from its source (a reborrow, not an exposed-then-reconstructed pointer).
  5. Frame-limited reuse tokens: a reuse token is a freed-but-not-returned cell reused by a same-size alloc; it must carry provenance from the allocation (the reuse path returns a pointer with valid provenance for the new object, not a re-cast of the old integer).

CI GATES:
  - A Miri job (cargo miri test under strict-provenance, -Zmiri-strict-provenance) over the object/RC/GC core and a cycle-collection unit test corpus. It must pass on the NaN-box pack/unpack, inc/dec, the registry, deduce_unreachable, molt_traverse/molt_clear, and the resurrection re-check.
  - Prove the gate FAILS on a synthetic `ptr as usize` round-trip introduced into the registry (so the gate is known-effective, not theater).
  - The existing rc==0 underflow-abort and resurrection-abort stay as runtime tripwires; the Miri gate is the static complement.

INTERACTION WITH GIL: collection is stop-the-world under the GIL, so the registry and gc_refs scratch need no atomics; but the GIL-held assumption must be asserted at collector entry (with_gil_entry), and the registry must be per-runtime-instance (not process-global) to stay sound under Luau embedding / multiple runtimes.

MULTI-BACKEND PLAN — native (Cranelift) / WASM / LLVM / Luau.

PRINCIPLE (CLAUDE.md "optimizations are IR facts, not native codegen"): every memory-management fact lives in portable TIR and is consumed identically by all backends; only two things are backend-conditional, both documented.

EXPLICIT NON-GOALS: no MMTk binding, no precise stack maps for a tracing GC, no per-backend GC-root implementation. These would regress perf, break __del__ parity, and multiply backend complexity 4× with a WASM hole.

## How it fixes both failure classes

HOW THIS ELIMINATES BOTH FAILURE CLASSES.

DICT/ACYCLIC LEAK CLASS (the 513MB `for i in range(1e6): d={...}` leak). Root cause: the Perceus mutable-slot rule `(:=) ≡ read-old; overwrite; drop-old` is not discharged when the slot value is not a visible owned SSA resource. Concretely a loop-carried OWNED block-arg phi that is dead on the back-edge (pure reassignment) is dropped by NONE of the existing rules — §1 keys on operand uses (the body never reads the old d), §1b on op.results (a phi is a block arg), §3 edge-dying explicitly skips block args (drop_insertion.rs:2921), §2.7 is a documented NO-OP (2992-3002), §5 only adds IncRefs — so only the final value is released at loop exit and the N-1 intermediates leak. The runtime free path is correct (explicit `del d`, which lowers to a per-iteration DecRef of the old slot value, flattens RSS to 15MB), proving the gap is purely the missing implicit drop-old.
  THE FIX makes "a slot/dict store of an owned value is a drop obligation discharged exactly once on every path including back-edges" a first-class ownership-lattice invariant: Phase 1 emits the back-edge DecRef for dead loop-carried phis, unifies the module-dict and boxed/async lanes into the same TIR obligation (retiring the legacy backend lane atomically), and extends to finalizer-sensitive slots. Phase 2's consume-exactly-once verifier then proves every owned root has exactly one release on every path, making the class UNEXPRESSIBLE — a zero-release-on-a-path is a hard verifier error, not a silent leak. Phase 3's activation completeness extends the same guarantee to try/except and generator/async bodies, where the pass currently bails and leaks ALL temporaries (the largest blast radius). A Perceus-correct pass cannot miss this because in Perceus the rebound value is in Gamma and the back-edge/[smatch] machinery drops exactly the owned references each path does not forward; the only precondition is that the resource is visible as owned — which is exactly the precondition Phase 1 restores across all non-SSA-phi lanes.

CYCLE LEAK CLASS (every reference cycle leaks forever; gc.collect() reclaims zero). Root cause: molt_gc_collect (ops.rs:1810) is a stub that only clears weakrefs-to-already-dead and resets counters — no graph traversal, no trial deletion; tp_traverse/tp_clear are inert C-ABI fields. Pure RC structurally cannot reclaim a self-sustaining group (a.peer=b; b.peer=a both dropped stay at rc≥1).
  THE FIX builds Tier 2: Phase 4 extracts the real molt_traverse/molt_clear from the dealloc cascade (so the collector sees exactly the children the deallocator frees — no drift), Phase 5 tracks cycle-capable (non-GREEN) containers in a side registry, Phase 6 runs CPython's deduce_unreachable (which computes "objects whose refcount is fully explained by internal edges" = the cycle garbage) and frees it via molt_clear's in-place ref-drop, collapsing the cycle through the normal RC cascade. The acyclic majority is untouched because GREEN/atomic objects are never registered and the collector only runs on threshold — so this adds CPython-parity cycle reclamation without taxing the Perceus fast path. CPython-identical observable behavior follows from reusing molt's existing finalizer/weakref machinery in CPython's exact 6-step order with the mandatory resurrection re-check.

WHY THE TWO FIXES ARE KEPT SEPARATE (per zero-workaround policy): the acyclic leak is a compile-time RC defect and MUST be fixed in the drop pass; using the cycle collector to mop it up would mask a Perceus bug with a slow scan — a forbidden workaround and a perf regression (RC should free it promptly and for free). Phase 1 precedes and is independent of Phases 4-6.

- NON-MOVING FREE-LIST HEAP vs THE CITED RC EVIDENCE — the design's own headline perf justification is self-contradicting. This is false for Python dict/list semantics. Perceus reuse (`if is_unique then reuse-in-place else copy`) fires ONLY when the object is provably UNIQUE (rc==1) AND same-size-class at the reuse point. In `for i in range(1e6): d = {'k':i,'v':[i,i+1,i+2]}` the dict literal is FRESHLY ALLOCATED each iteration by BuildDict and the prior `d` is being dropped — there is no `d` whose backing store is being reused in place; reuse analysis matches a CONSTRUCTOR consuming a DROPPED value of the same shape, which a dict-literal lowering does not expose. Worse, Python dict/list are runtime-resizable hash/vector containers with interior heap (the `[i,i+1,i+2]` list), so 'reuse' would have to reuse BOTH the dict header AND the nested list across iterations with matching capacity — Koka gets FBIP reuse on FIXED-LAYOUT algebraic constructors, not on amorphous resizable Python containers. The perf model oversells the steady state: the realistic outcome is O(1) peak RSS but O(N) alloc/free churn, which on the non-moving free-list (flaw 1) is exactly where molt is most exposed vs Codon's bump-allocated typed lowering. **Serious risks:**
- IMMORTAL RC IS NOT 'ZERO COST' — the perf model lists immortal (None/True/small-int/interned/type/code) as '0 op AND 0 cache-line invalidation'. For a hot loop hammering small-int/None operands in BOXED form this is a per-op load+branch that CPython's PEP-683 also pays, so it's not a LOSS, but the model's '0 cache-line invalidation' is wrong: a read still pulls the line into shared state, and under the GIL it's cheap but non-zero. The claim 'reuse so cyclic and acyclic finalization are byte-identical' is the root error: byte-identical finalization requires DIFFERENT drivers around a SHARED __del__-invocation primitive.

**Fatal flaws:**
- DESTRUCTION ORDER IS INVERTED vs CPython 3.12 (the single most parity-sensitive contract, and the design gets it backwards). The design's cycle_collector_design step list runs: 1. handle_weakrefs FIRST, 2. move_legacy_finalizers, 3. finalize_garbage. CPython 3.12's gc_collect_main runs the OPPOSITE relative order of the first two phases: move_legacy_finalizers + move_legacy_finalizer_reachable run BEFORE handle_weakrefs, which runs before finalize_garbage. Verified against v3.12.0 Modules/gcmodule.c: the literal call sequence is deduce_unreachable -> untrack_tuples -> gc_list_merge -> move_legacy_finalizers -> move_legacy_finalizer_reachable -> handle_weakrefs -> finalize_garbage -> handle_resurrected_objects -> delete_garbage. The design even claims (verbatim in cycle_collector_design) 'Weakref clearing STRICTLY PRECEDES finalizers' as the headline invariant while simultaneously placing legacy-finalizer handling AFTER weakrefs. This is a self-inconsistent and observably wrong ordering. Although molt asserts it has no legacy tp_del finalizers (so move_legacy_finalizers nominally collapses), the ORDER STILL MATTERS because move_legacy_finalizer_reachable PULLS objects reachable-from-finalizers OUT of the unreachable set BEFORE handle_weakrefs decides which weakrefs to clear/fire. Getting the order wrong changes which weakrefs observe None and which callbacks fire. This is a fatal observable divergence as specified.
- molt's EXISTING weakref machinery (weakref_clear_for_ptr) is INCOMPATIBLE with CPython's two-pass batched weakref protocol the collector REQUIRES, and the design mandates 'REUSE weakref_clear_for_ptr' verbatim. I read weakref.rs:17-58: weakref_clear_for_ptr clears and then IMMEDIATELY INVOKES the callback for a SINGLE target's weakrefs, per-target. CPython's handle_weakrefs does the opposite and it is observably load-bearing: it FIRST clears ALL weakrefs pointing into the ENTIRE unreachable set (so every callback, when it runs, sees every dying object already reading None), THEN in a SEPARATE second pass invokes the surviving callbacks. If molt drives cyclic collection by calling weakref_clear_for_ptr per object in list order, then callback A (fired while clearing object 1) can still observe object 2 (also dying, in the same cycle) via a weakref that has NOT yet been cleared -> CPython guarantees it reads None, molt would expose a live-looking dying object or fire in the wrong order. The design's 'reuse the existing machinery so cyclic and acyclic finalization are byte-identical' is therefore FALSE for the cyclic path: the existing per-target clear-then-call helper cannot produce CPython's all-clear-then-all-call semantics. A new batched two-pass weakref driver is required for the collector; the acyclic dec_ref path can keep the per-object helper, but they are NOT the same function.

**Serious risks:**
- WEAKREF CALLBACK DROP-RULE for self-collecting weakrefs is unstated and easy to get wrong. CPython's handle_weakrefs DROPS the callback (does not enqueue it) when the WEAKREF OBJECT ITSELF is in the collecting/unreachable set (gc_is_collecting(AS_GC(wr)) -> continue, asserting wr_object==None). The design mentions this once ('a weakref unreachable in the same cycle has its callback DROPPED') but molt's weakref entries live in a SIDE registry keyed by PtrSlot, NOT as gc-tracked container objects in the candidate set. So molt has no direct analogue of gc_is_collecting(wr): it must independently determine whether the weakref OBJECT is itself in the unreachable set being collected and suppress its callback. Without an explicit mapping from registry entry -> is-the-weakref-itself-unreachable, molt will FIRE callbacks CPython drops. Concrete divergence: a cycle containing both an object and a weakref-with-callback to a sibling in the same cycle -> CPython drops that callback, molt would fire it.
- weakref_resolve_target_ptr uses a NON-CPYTHON rc<=2 HEURISTIC to declare a target dead (weakref.rs:199: 'if rc <= 2 && !target_bound_in_module_globals -> treat as dead, clear, drop callback'). This is an approximation of liveness, not refcount-exact CPython weakref semantics. Under the new precise-RC + collector regime this heuristic becomes a correctness hazard: it can clear a weakref (and permanently drop its callback) for an object that is actually still live but transiently at rc<=2, and it can mask the very leak/collector-timing the design is trying to make exact. It also means weakref-reads-None timing diverges from CPython (which is purely rc==0 / on-collection). This must be deleted, not carried forward, or the 'byte-identical weakref clearing order' claim is false on the acyclic path too.

The collector's finalize_garbage must NOT route through dec_ref_ptr's rc==0 path (the objects are on the unreachable list with internal references still intact, so their rc is NOT 0). If the collector calls maybe_run_object_finalizer, that function's inc/dec self-balancing and its rc>1 'resurrected' test were designed for the rc==0 acyclic context; in a cycle every member has rc>=1 from its peers, so prev>1 will ALWAYS be true and EVERY finalized cyclic object will be misclassified as resurrected. The design says 'reuse maybe_run_object_finalizer (NOT a second finalizer path)' (cycle_collector_design step 3) but that exact reuse is unsound in the cyclic context. The two finalizer entrypoints share __del__ invocation but MUST NOT share the resurrection decision.

The design says 'deterministic GC-list (insertion) order, never address/hash order' (cycle_collector_design step 3). Since __del__ ordering within a cycle is observable (and tested in CPython's own suite), molt must replicate the gc_list construction order exactly, not 'insertion order'. As written this is a likely divergence.
- FINALIZATION-DURING-INTERPRETER-SHUTDOWN parity is entirely unaddressed. CPython at shutdown does gc.collect() passes plus _PyGC_CollectNoFail and finalizes modules/clears __main__, and __del__ ordering/whether-it-runs at shutdown is famously observable (and version-specific). The design's migration_path and cycle_collector_design never specify molt's shutdown sequence: does molt run a final cycle collection at exit? - gc.collect() RETURN VALUE semantics are mis-specified. The design says 'gc.collect() returns n+m (n=uncollectable=0 in molt, m=collected)'. Verified against gcmodule.c: m (collected) = handle_weakrefs return (weakrefs cleared whose refcount hit zero) PLUS gc_list_size(final_unreachable); n (uncollectable) = objects moved to gc.garbage (legacy only). The design must pin EXACTLY what 3.12's gc.collect() returns (it is the sum reported to the 'stop' callback's 'collected' field semantics) and reproduce the weakref-cleared contribution to m, which the design's 'm=collected' gloss omits. Getting the integer return wrong is a trivially-tested divergence.
- gc.get_objects() / get_referrers ORDER and CONTENT parity is under-specified and likely divergent. The design builds these from 'walk molt_traverse + registry'. Any object molt treats as GREEN that CPython still tracks (CPython only untracks tuples/dicts proven atomic AT COLLECTION TIME, not at alloc) yields gc.get_objects()/is_tracked divergence. In RELEASE (#[cfg(not(debug_assertions))], the ship profile for WASM/LLVM/Luau), obj-model lib.rs:235-240 boxes via `ptr as u64` + POINTER_MASK — a raw cast, no expose — and as_ptr (lib.rs:318-320) recovers via with_exposed_provenance_mut. expose_provenance runs only in debug register_ptr. with_exposed_provenance requires a prior matching expose; release from_ptr never exposes, and the integer is masked to 48 bits + sign-re-extended, so no provenance is recoverable. sanitizers.yml:24-26/106-108 says verbatim strict-provenance is 'intentionally NOT enabled' because the boxing uses exposed provenance. The fix (expose_provenance on every profile — a codegen no-op) is cheap but the design must NAME it; as written it claims a property the binary lacks and proposes a gate that cannot pass.

FATAL #2 — gc_refs CORRUPTS VIA SHARED COLD HEADERS. The design stores transient per-object gc_refs in 'a cold-header slab field via cold_idx.' mod.rs:640-660 documents SHARED_COLD_IDX_BIT (1<<31): many instances of one class share ONE cold header. Per-object gc_refs in a shared header means update_refs/subtract_refs for instance A clobbers every sibling's gc_refs, breaking CPython's 'gc_refs==0 iff unreachable' invariant => an externally-rooted object stomped to 0 is freed while live (UAF, council P0). The design never references SHARED_COLD_IDX_BIT. Fix: use the design's own alternative, a transient per-collection HashMap<addr,isize> (stop-the-world under GIL, no atomics, no header growth, O(N_candidate)).

SERIOUS RISKS. (A) POINTER_MASK truncates to 48 bits + sign-extends from bit 47; on LA57/AArch64-LVA a real pointer reconstructs WRONG, and the collector dereferences these across the heap (wild write) — needs a <=47-bit arena invariant + startup assert. (B) The traverse==dealloc 'pure refactor' gate ignores that dealloc is ORDER-CRITICAL (TYPE_ID_TYPE arm: payload-before-dict to avoid double-free); a set-equality gate cannot preserve ordering — needs a two-consumer contract over one generated ordered child list. (C) molt-check's consume-exactly-once proves multiplicity, not no-use-after-release on unwind edges — Phase 3 (exception-CFG dominance, the design's own highest-risk step) needs the post-dominance property too. (D) Luau cross-boundary cycle ownership is asserted-fixed but not proven decoupled from the host tracing GC; straddling cycles may be collectable by neither — make it a proven boundary or a loud xfail. (E) Phase 7 cursor/reuse add dangling-ptr-in-f64 hazards whose only catch-net is the strict-provenance gate that FATAL #1 shows does not exist — sequence Phase 7 strictly after the gate is real.

BOTTOM LINE. Keep the architecture. Before any 'world-class / strict-provenance-clean' claim: (1) make release from_ptr call expose_provenance and wire+prove the strict-provenance gate; (2) move gc_refs to a transient map; (3) pin the heap to <=47-bit VA; (4) give the keystone a two-consumer ordering contract; (5) extend molt-check to no-use-after-release before Phase 3; (6) prove or xfail the Luau boundary. Items 1-2 are fatal-as-written; all six are concretely fixable within the design's own stated machinery. I attacked the SAFETY/PROVENANCE/BACKEND story hard and found two real fatal flaws plus four serious risks — this is not a rubber stamp, but the underlying architecture survives once these are repaired.

The ONLY real constraint is the cpython-abi C-extension hooks + signal handlers cited in refcount.rs:4 — these must be gated: keep atomic RC only in the (rare, opt-in) configuration that loads foreign C extensions or when GIL is released for @par. The Cranelift block-fragmentation bug (simple_backend.rs:1379) is the real blocker; the structurally correct fix is a single-basic-block, branchless inline sequence (the immortal check is already branchless via `select`; do the same for the is_ptr tag test where the value is statically known to be a pointer the drop pass can often prove, eliminating the brif entirely). REJECT for the steady state: leaving RC as opaque atomic C-ABI calls (the current default) — it violates the performance contract on every native benchmark and is the dominant non-algorithmic overhead in boxed-value loops.

WHY (decisive trade-offs, in priority order):

### Deep code map of molt's current RC+GC architecture + IR-level pinpoint of the confirmed acyclic loop-local leak (and the cycle leak)
ADOPT the structural fix; do NOT patch around it.

ACYCLIC LEAK (one bug CLASS, fix in TIR): the §2.7 loop-carried rule in drop_insertion.rs:2992-3002 is a documented NO-OP that wrongly assumes §1/§1b/§2.5 cover the loop-carried phi. They do not when the back-edge OVERWRITES the local without READING it (pure reassignment `d = expr` where expr doesn't reference d). The header phi is then dead-on-back-edge: §1 (last_use=operand uses) misses it, §1b (op.results only) misses it (a phi is a block arg), §2.5 edge-dying EXPLICITLY skips block args (drop_insertion.rs:2921), and §5 only adds IncRefs. So the previous-iteration value is structurally undroppable; only the FINAL value is released at loop exit. Implement §2.7 as actually specified: for each back-edge `→ header(new_val_pos=p)`, the previous value is `header.args[p]` (the phi); if it is owned/droppable and not used after the point new_val is computed in the latch, insert DecRef(phi) just before the back-edge branch (CPython STORE_FAST-on-overwrite semantics). This is the rung-3 "ordered release obligation" on the ownership lattice (doc 55), not a placement special-case. The accumulator shape (`total = total + x`) already works ONLY because the phi is READ in the body and dropped at that read; the fix must not double-drop it (guard: only place the back-edge drop when the phi is NOT read between header-entry and the latch, i.e. dead at the back-edge).

This is a FAMILY: it bites any loop-local owned variable reassigned without an intervening read in the same iteration — confirmed-prone shapes: plain reassign `x = f()`, subscript/attr overwrite where the container is the loop-local being rebound, and any rebind whose RHS doesn't read the old binding. It does NOT bite read-modify accumulators or loop-invariant containers (`lst` defined outside the loop with `lst.append(...)`). The fix is one rule covering the whole class.

CYCLE LEAK (separate, larger arc — REJECT the stub as a long-term answer): molt_gc_collect must become a real CPython-parity cyclic collector (mark-sweep over tracked container objects with trial deletion / refcount subtraction), wired to actual tp_traverse over the per-type inner-ref sets the dealloc cascade already enumerates. This is the acknowledged-missing "fifth boundary class" of doc 55 (line 645). Until it lands, `gc.collect()` returning 0 for cycles is a hard parity violation.
