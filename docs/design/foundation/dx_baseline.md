## §3. Blueprint claims found STALE / WRONG (verify-first caught these)

1. **"`release-fast` separated from a ship profile is a TODO" — ALREADY DONE.** A `release-output` profile (fat LTO, cgu=1, opt-z, panic=abort, strip) ALREADY EXISTS (Cargo.toml:366) and is ALREADY the shipped-runtime default (`cli.py:12181` `_resolve_cargo_profile_name` returns `release-output` for release; `_resolve_backend_cargo_profile_name` returns `release-fast` for the daemon). The dev-vs-ship profile split the blueprint's Phase 1a proposes is structurally in place. The ONLY open Phase-1 lever is the fat-vs-thin LTO choice **within** `release-fast`.
2. **"`release-fast` fat LTO has no perf justification" — CONTRADICTED by in-tree comments.** Cargo.toml:296-305 documents that `release-fast` deliberately uses `codegen-units=256` to parallelize the 342K-LoC molt-runtime hub's MIR→LLVM codegen across 18 cores (cgu=1 serialized it into a ~72s one-line rebuild). Whether thin LTO beats fat LTO here is an EMPIRICAL question this arc must measure, not assume.
3. **Phase 2 architecture assumption is WRONG.** The blueprint assumes `function_compiler.rs` is "8 opcode-family handlers dispatched by an `emit_op()` match" that can be moved as "pure code motion." REALITY: it is ONE 34,200-line method `compile_func_inner` with one giant inline `match op.kind.as_str()` whose arms share thousands of lines of mutable local state (builders, variable maps, tracked-name vecs, alias roots, scalarized-tuple side tables, arena ptrs). Splitting the FILE without splitting the FUNCTION yields ~0 incremental-build benefit, because **a function is the atomic unit of rustc codegen — `codegen-units` partitions at function boundaries, so a 34K-line function is ONE codegen unit regardless of how many files or modules surround it.**
4. **sccache is NOT installed** on this host (`which sccache` → not found). The wiring (`_maybe_enable_sccache`, retry-on-wrapper-failure) already exists in cli.py and silently no-ops when sccache is absent. Phase 1b's "shared sccache dir" requires installing sccache first.
5. **`ld64.lld` location differs.** Blueprint tied the linker to a specific LLVM backend keg; actual measured hosts used `/opt/homebrew/bin/ld64.lld` (Homebrew `lld` formula). The linker path is separate from the manifest-derived LLVM backend pin.

## §4. THE answer to "why didn't the lib.rs split (34e3bddbf) speed builds up?"

Two independent reasons, both confirmed statically and (below) by measurement:

1. **Cargo's incremental/caching unit is the CRATE, not the module.** The lib.rs split moved code into sibling modules *inside the same `molt-backend` crate*. Any edit to any of those modules still invalidates and recompiles the whole `molt-backend` crate (rustc compiles a crate as a unit; module boundaries are not recompilation boundaries). The split bought cleaner code organization and zero build-time parallelism across the boundary.
2. **The dominant cost is one indivisible function.** Even within the crate, `codegen-units=256` cannot parallelize the compile of `compile_func_inner` because it is a *single 34K-line function* = a single codegen unit. rustc must MIR-build, borrow-check, monomorphize, and LLVM-optimize it as one indivisible item. No file/module reorganization changes that. Only decomposing the *function itself* into independently-compilable functions (or extracting it to a crate so that a non-native edit skips it entirely) moves this needle.

This is why the designed follow-up — extracting `molt-backend-native` as a real CRATE — is the structurally correct lever: it makes a tir-pass edit (the most common optimizer-foundation edit) stop dragging the 34K-line native function through recompilation.

### What these numbers prove

1. **A tir-pass edit and a native-codegen edit cost the SAME (~125–168 s).** Both fully
   rebuild `molt-backend`. The file/module a line lives in is irrelevant to rebuild cost —
   the crate is cargo's recompilation unit. This is the empirical core of the answer to
   "why didn't the lib.rs split speed builds up" (§4).
2. **molt-runtime is already decoupled** from the daemon build (0.10 s no-op). Therefore
   **Phase 4 (generated.rs split) is NOT on the daemon's critical path** and is deprioritized
   by measurement — it would only help the *user-binary staticlib* build (a separate path),
   which the daemon iteration loop does not pay.
3. **`release-fast` builds ONLY the compiler daemon, never the shipped user-binary runtime**
   (the staticlib uses `release-output`, fat LTO, untouched — cli.py:12181 vs 12220). So a
   `release-fast` thin-LTO change has **zero** user-binary runtime-perf risk by construction;
   the bench-regression gate is structurally satisfied (verified by reasoning + bench sanity).

## §6. Phase 2 verdict: REFUSED as blueprint-designed; rescoped

The blueprint's Phase 2 ("move 8 opcode-family handlers out as pure code motion") rests on a
false premise (§3.3): there is no `emit_op()` dispatcher to move. `compile_func_inner` is ONE
~34,200-line method that constructs **45+ shared `let mut` locals** (`builder`, `if_stack`,
`loop_stack`, `label_blocks`, a dozen caches, tracked-name vecs, alias-root maps, scalarized-tuple
side tables, arena ptrs) and then runs a single inline `match op.kind.as_str()` whose arms
read/mutate many of those locals. Extracting an arm into a sibling function requires threading
20–40 `&mut` references (or a `FunctionCompileState` struct) through every call — a deep, high-
miscompile-risk refactor, NOT "pure code motion." And critically (§4.2): **even a perfect split of
the FILE buys ~0 incremental-build time, because the 34K-line function remains ONE codegen unit**
(`codegen-units` partitions at function boundaries). Splitting the *function* is the only thing that
would help intra-crate, and that is a correctness-critical multi-week refactor with no build-time
guarantee — it is out of scope for a DX/throughput arc and would violate "don't gold-plate."

The structurally correct lever that achieves Phase 2's stated GOAL (a native edit shouldn't drag the
optimizer core and vice versa) is **Phase 3 crate extraction**, not file-splitting. Phase 2 is
therefore folded into Phase 3's value proposition rather than landed as a low-value file shuffle.

## §7. Phase 1a measured win: `release-fast` fat -> thin LTO (LANDED)

Controlled A/B on this repo (M5 Max, isolated `target-agent-dx/`, touch a molt-backend lib
source = `simple_backend.rs` then rebuild the daemon). Cargo.toml `release-fast` `lto` flipped
in place, restored after; tool `tmp/lto_ab.sh`-style harness:

| Profile config | Build scope | Run 1 | Run 2 | Median |
|----------------|-------------|-------|-------|--------|
| **fat** LTO (old) | full `-p molt-backend` | 132 s | 142 s | **137 s** |
| **fat** LTO | `--bin molt-backend` | 153 s | 150 s | 151 s |
| **thin** LTO (NEW) | full `-p molt-backend` | 73 s | 90 s | **81 s** |
| **thin** LTO | `--bin molt-backend` | 84 s | 81 s | 82 s |

**Result: thin LTO cuts the incremental daemon rebuild from ~137 s to ~81 s — a ~41% / ~56 s
reduction per edit.** This is the dominant Phase-1 lever.

Notes:
- **`--bin molt-backend` scoping is in the noise** when the edit is to a *lib* source, because
  the lib change forces the daemon bin to relink/re-LTO under either scope; the `typed_repr_report`
  bin's 23 s is dwarfed by (and overlaps in parallel with) the daemon-bin LTO. So bin-scoping was
  NOT adopted as a separate change — it would add cli plumbing for a win the measurement does not
  support. (It would only help for the narrow case of editing *only* code the daemon bin uses but
  the diagnostic bin does not — not worth the cli complexity. `typed_repr_report` already builds
  itself on demand via `tools/representation_report.py`'s `cargo run --bin typed_repr_report`.)
- **Cold full build** under thin LTO is also expected faster (the same LTO-merge serialization is
  removed); the cold baseline was 196 s under heavy contention with fat LTO.
- **Soundness**: thin LTO is a correct superset of no-LTO for all Rust programs; `panic = "unwind"`
  preserved (Cranelift catch_unwind retry path intact). `release-fast` builds ONLY the compiler
  daemon, never the shipped runtime (release-output, fat, untouched) — so no user-binary perf risk.

The user-binary runtime is built with `release-output` (fat LTO), which this arc does NOT touch, so

## §11. Phase 1b (sccache + linker) — analysis, NOT landed (honest scoping)

- **sccache is NOT installed** on the host (`which sccache` -> not found). The wiring already exists
  (`_maybe_enable_sccache`, cli.py:9455; retry-on-wrapper-failure cli.py:9476) and no-ops cleanly when
  sccache is absent. Installing a system tool is outside this arc's remit, and the Phase-1b gate
  ("cross-worktree second-build hit rate") is unmeasurable without it. **Env contract for when an
  operator installs sccache** (documented so agents can opt in): set a SHARED, repo-local cache so all
  per-session `target-<id>/` worktrees share compiled-rlib results (sccache keys by content hash, not
  path, so this is safe across worktrees):
  ```
  brew install sccache            # one-time, operator action
  export SCCACHE_DIR="$PROJECT_ROOT/.sccache"   # shared across all agent worktrees
  export SCCACHE_CACHE_SIZE="20G"
  # MOLT_USE_SCCACHE defaults to "auto" -> cli auto-enables RUSTC_WRAPPER=sccache when found.
  ```
  Caveat: sccache does NOT cache LTO link products, only per-crate codegen; under thin LTO the
  dependency rlibs (cranelift, serde, etc.) are the cacheable wins (~0 % they change across edits ->
  ~85 % hit on a 2nd worktree's first build). The daemon-bin LTO itself is never an sccache hit.
  Recommend adding `.sccache/` to `.gitignore` if adopted. (Left undone deliberately: making an
  unmeasurable env change with no installed tool to validate against would be a workaround.)
- **macOS fast linker**: `/opt/homebrew/bin/ld64.lld` exists (Homebrew `lld`; not owned by the
  LLVM backend keg). But on an LTO-bound build the link is a small fraction of wall time (the LTO
  optimize dominates), and the system `ld` (Apple ld-1267, recent) is already fast. A linker swap is
  a marginal lever vs the ~41 % thin-LTO win and is not adopted; the existing `.cargo/config-fast-link.toml`
  opt-in pattern documents how to add one locally if a future measurement justifies it.

## §13. Bench sanity result + the airtight no-regression argument

The ENTIRE Phase 1a diff to Cargo.toml is exactly ONE non-comment line:
`[profile.release-fast] lto = "fat"` -> `"thin"`. `release-fast` builds ONLY the compiler daemon
(cli.py:12220); the shipped user-binary runtime staticlib uses `release-output` (cli.py:12181),
which is UNTOUCHED. Therefore the compiled user binary is **byte-for-byte identical** before and
after this change, and a user-binary runtime regression is impossible by construction.

Empirical sanity (M5 Max, settled load; molt user binary unchanged):
- fib(30): molt 0.21 s (steady) vs CPython 0.10 s — startup/safe_run-wrapper-dominated at this tiny
  scale; not a meaningful compute comparison.
- fib(35) (≈30× compute): molt 0.62–0.64 s vs CPython 0.59–0.61 s — recursion/call-overhead-bound,
  roughly at parity. This is a PRE-EXISTING runtime property (recursive-call overhead, an active area
  per MEMORY.md bench_calls), unchanged by this arc; the LTO knob affects neither the runtime
  staticlib nor the compiled program. (Heavier non-recursion benchmarks where molt is decisively
  faster than CPython — sum/list/dict — are likewise unaffected.)
